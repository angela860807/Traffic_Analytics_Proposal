## v4_precursor_eval

- Date: 2026-06-15
- Goal: Check whether early detection is possible without retraining when the evaluation fault data includes a pre-fault precursor window.
- Baseline being compared: v3 LSTM-AE with `thr999`.
- Training data: unchanged `data/normal_samples3.csv`.
- Evaluation data: new `data/fault_samples4.csv`.
- Model: unchanged `model/lstm_ae_v3_thr999.pt`.
- Scaler: unchanged `model/scaler_v3.pkl`.
- Threshold: unchanged `model/threshold_v3_thr999.json`.
- Warning threshold: `0.429772`.
- Critical threshold: `0.572504`.
- Generated by: `generate_fault_samples4_precursor.py`.
- Evaluated by: `evaluate_lstm_ae_v4_precursor.py`.

### Data Change

- Original v3 fault pattern: `NORMAL 120m -> FAULT 240m -> RECOVERY 120m`.
- v4 fault pattern: `NORMAL 60m -> PRECURSOR 60m -> FAULT 240m -> RECOVERY 120m`.
- `is_fault` remains true only for the actual fault window.
- `risk_label` is true for `PRECURSOR + FAULT`.
- `scenario_key` is used to keep each scenario contiguous during evaluation.

### Result

- Scenario count: `24`.
- Detected count: `24`.
- Early detection count: `19`.
- Late detection count: `5`.
- Average early lead time: `45.8 min`.
- Max early lead time: `59 min`.
- Average late detection delay: `109.2 min`.
- Max late detection delay: `141 min`.
- Average F1: `0.8365`.
- Best scenario: `NETWORK_INSTABILITY_WEAK`.
- Worst scenario: `OCR_QUALITY_WEAK`.

### Interpretation

- With precursor-aware evaluation data, the existing v3 model and fixed 99.9% threshold can detect many fault scenarios before the actual fault window starts.
- The improvement is caused by changing the evaluation data definition, not by retraining or lowering the threshold.
- Strong early-detection results appear for camera offline, FPS degradation, frame drop, latency degradation, resource saturation, and network instability.
- Blur and OCR quality scenarios remain weak, especially `OCR_QUALITY_WEAK`; these should be handled together with rule, robust z-score, trend projection, and OCR sample-count policies.

- `data/fault_samples4.csv`
- `results/lead_time_report_v4.csv`
- `results/scenario_metrics_v4.csv`
- `results/evaluation_summary_v4.json`

## v5_timepattern

- Date: 2026-06-15
- Goal: Check the effect of richer normal time patterns on threshold stability and model performance.
- Method: Method 3, normal time-pattern diversity.
- Training data: new `data/normal_samples5_timepattern.csv`.
- Evaluation data: new `data/fault_samples5_timepattern.csv`.
- Model: new `model/lstm_ae_v5_timepattern_thr999.pt`.
- Scaler: new `model/scaler_v5_timepattern.pkl`.
- Threshold: new `model/threshold_v5_timepattern_thr999.json`.
- Warning threshold: `0.410734`.
- Critical threshold: `0.462065`.
- Generated by: `generate_simulation_data5_timepattern.py`.
- Trained by: `train_lstm_ae_v5_timepattern.py`.
- Evaluated by: `evaluate_lstm_ae_v5_timepattern.py`.

### Data Change

- Normal data keeps the same row count as v3: `60,480`.
- Fault data keeps the same row count as v3: `12,960`.
- Added richer normal variation:
  - morning/evening peak load,
  - weekend dampening,
  - camera-specific load offsets,
  - smooth 14-day daily variation,
  - night/rush-hour effects on blur, OCR, CPU, latency, and RTT.
- No precursor window was added in v5.

### Training Result

- Precision: `0.8119`.
- Recall: `0.7988`.
- F1: `0.8053`.
- ROC-AUC: `0.9696`.
- False alarm count: `61 / 60,451 normal sequences`.

### Scenario Evaluation Result

- Scenario count: `24`.
- Detected count: `24`.
- Early detection count: `0`.
- Late detection count: `24`.
- Average early lead time: `0 min`.
- Max early lead time: `0 min`.
- Average late detection delay: `42.5 min`.
- Max late detection delay: `196 min`.
- Average scenario F1: `0.4426`.
- Best scenario: `NETWORK_INSTABILITY_WEAK`.
- Worst scenario: `EXTERNAL_TRAFFIC_NORMAL_DEVICE`.

### Interpretation

- Normal time-pattern diversity alone does not create early detection because the fault evaluation data still has no pre-fault precursor window.
- The training-level full evaluation remained acceptable (`F1=0.8053`) while preserving the same false alarm count as v3 (`61`).
- Scenario-level F1 dropped because each scenario evaluation still combines one fault type with many normal periods, and the richer normal variation creates a harder discrimination problem.
- v5 is useful as a realism/stability experiment, not as the final predictive-maintenance result.
- The next important experiment is v6: combine v5 time-pattern normal data with v4-style precursor-aware fault data, then retrain and evaluate.

## v6_timepattern_precursor

- Date: 2026-06-15
- Goal: Combine richer normal time patterns with a pre-fault precursor window and evaluate the final candidate dataset/model.
- Method: Method 3 + Method 4.
- Training data: new `data/normal_samples6_timepattern_precursor.csv`.
- Evaluation data: new `data/fault_samples6_timepattern_precursor.csv`.
- Model: new `model/lstm_ae_v6_timepattern_precursor_thr999.pt`.
- Scaler: new `model/scaler_v6_timepattern_precursor.pkl`.
- Threshold: new `model/threshold_v6_timepattern_precursor_thr999.json`.
- Warning threshold: `0.410734`.
- Critical threshold: `0.462065`.
- Generated by: `generate_simulation_data6_timepattern_precursor.py`.
- Trained by: `train_lstm_ae_v6_timepattern_precursor.py`.
- Evaluated by: `evaluate_lstm_ae_v6_timepattern_precursor.py`.

### Data Change

- Normal data uses the same richer time-pattern generation as v5.
- Fault data uses a v4-style precursor structure:
  - `NORMAL 60m -> PRECURSOR 60m -> FAULT 240m -> RECOVERY 120m`.
- `risk_label` is true for `PRECURSOR + FAULT`.
- Generated row counts:
  - normal: `60,480`.
  - fault: `12,960`.
  - all: `73,440`.
- Fault phase distribution:
  - `FAULT`: `5,760`.
  - `RECOVERY`: `3,240`.
  - `NORMAL`: `1,800`.
  - `PRECURSOR`: `1,440`.
  - `EXTERNAL_TRAFFIC`: `720`.

### Training Result

- Precision: `0.7228`.
- Recall: `0.8509`.
- F1: `0.7816`.
- ROC-AUC: `0.9746`.
- False alarm count: `61 / 60,451 normal sequences`.

### Scenario Evaluation Result

- Scenario count: `24`.
- Detected count: `24`.
- Early detection count: `19`.
- Late detection count: `5`.
- Average early lead time: `43.4 min`.
- Max early lead time: `60 min`.
- Average late detection delay: `118.0 min`.
- Max late detection delay: `196 min`.
- Average scenario F1: `0.8051`.
- Best scenario: `NETWORK_INSTABILITY_MEDIUM`.
- Worst scenario: `OCR_QUALITY_WEAK`.

### Interpretation

- v6 is the strongest final-candidate experiment because it combines a more realistic normal baseline with precursor-aware predictive-maintenance evaluation.
- It preserves the same false alarm count as v3/v5 (`61`) while recovering early detection for `19 / 24` maintenance fault scenarios.
- Lead time is slightly lower than v4 (`45.8 -> 43.4 min`), which is expected because richer normal variation makes the model more conservative.
- Scenario F1 is lower than v4 (`0.8365 -> 0.8051`) but substantially higher than v5 (`0.4426`).
- Camera offline, FPS degradation, frame drop, latency degradation, resource saturation, and network instability are suitable for early warning with LSTM-AE shadow scoring.
- Blur and OCR quality remain weak, especially `OCR_QUALITY_WEAK`; these should be handled by rule/z-score/trend/OCR sample-count policies rather than relying on LSTM-AE alone.
