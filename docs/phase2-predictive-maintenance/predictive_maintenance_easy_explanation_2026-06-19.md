# 예지보전 기능 쉬운 설명 자료

작성일: 2026-06-19

## 1. 이 프로젝트를 한 문장으로 설명하면

이 프로젝트는 CCTV와 AI 교통 분석 시스템이 정상적으로 동작하는지 계속 확인하고, 문제가 생길 것 같으면 미리 알려주고 정비 티켓까지 만들어주는 시스템이다.

조금 더 쉽게 말하면:

> 차를 찍는 카메라와 AI 분석기가 잘 일하고 있는지 감시하는 관리자 시스템

기존 프로젝트가 차량을 탐지하고 과속을 확인하는 기능이었다면, 이번 프로젝트는 그 기능이 안정적으로 운영되도록 도와주는 기능이다.

## 2. 기존 프로젝트와 이번 프로젝트의 차이

기존 프로젝트:

- CCTV 영상에서 차량을 찾는다.
- 번호판 OCR을 한다.
- 차량 속도를 계산한다.
- 과속 여부를 탐지한다.

이번 프로젝트:

- 카메라 영상이 너무 흐려지지 않았는지 확인한다.
- 영상 FPS가 너무 낮아지지 않았는지 확인한다.
- 네트워크 지연이 심하지 않은지 확인한다.
- AI 분석 시스템이 정상적으로 데이터를 받고 있는지 확인한다.
- 이상이 심하면 정비 티켓을 자동으로 만든다.

즉, 기존 프로젝트가 “차량을 분석하는 기능”이라면, 이번 프로젝트는 “차량 분석 시스템 자체를 관리하는 기능”이다.

## 3. 왜 이런 기능이 필요한가

실제 도로 CCTV를 운영한다고 생각해보면, 차량을 잘 탐지하는 것만큼 중요한 문제가 있다.

예를 들어:

- 카메라 렌즈가 흐려질 수 있다.
- 영상이 끊길 수 있다.
- FPS가 떨어져서 차량 움직임을 제대로 못 볼 수 있다.
- 서버나 네트워크가 느려질 수 있다.
- OCR 정확도가 떨어질 수 있다.

이런 문제가 생기면 차량 탐지, 번호판 인식, 속도 계산 결과도 믿기 어려워진다.

그런데 관리자가 모든 카메라를 계속 눈으로 확인하는 것은 어렵다. 그래서 시스템이 스스로 상태를 확인하고, 위험한 카메라를 먼저 알려주는 기능이 필요하다.

## 4. 이 프로젝트의 핵심 흐름

전체 흐름은 다음 5단계로 설명하면 된다.

```text
영상 입력
  -> 카메라 상태 수집
  -> 이상 징후 감지
  -> 정비 티켓 생성
  -> 운영자 조치
```

각 단계를 쉽게 풀면 다음과 같다.

1. 영상 입력
   - CCTV 영상 또는 준비된 샘플 영상을 시스템에 넣는다.
   - 기존처럼 차량 탐지, OCR, 속도 계산이 동작한다.

2. 카메라 상태 수집
   - 시스템은 카메라가 얼마나 잘 동작하는지 상태값을 모은다.
   - 예: FPS, 프레임 드롭, 영상 흐림, 응답 지연, 네트워크 상태

3. 이상 징후 감지
   - 수집된 상태값이 기준보다 나쁘면 이상으로 판단한다.
   - 예: FPS가 너무 낮다, 영상이 너무 흐리다, 네트워크 지연이 크다

4. 정비 티켓 생성
   - 심각한 이상이면 사람이 조치할 수 있도록 정비 티켓을 자동 생성한다.
   - 티켓에는 어떤 카메라에 어떤 문제가 있는지 기록된다.

5. 운영자 조치
   - 운영자는 관리자 화면에서 이상 목록과 티켓을 확인한다.
   - 담당자를 배정하거나 상태를 변경하고, 조치가 끝나면 해결 처리한다.

## 5. 발표에서 가장 쉽게 말하는 방법

발표에서는 복잡한 기술명보다 운영 흐름을 중심으로 설명하는 것이 좋다.

추천 설명:

> 저희 시스템은 CCTV 영상에서 차량을 인식하는 것뿐 아니라, 그 CCTV와 AI 분석 시스템이 정상적으로 동작하는지도 함께 확인합니다. 영상이 흐려지거나 FPS가 떨어지거나 네트워크가 불안정하면 시스템이 이를 이상 징후로 감지하고, 심각한 경우 정비 티켓을 자동으로 생성합니다. 운영자는 관리자 화면에서 위험 카메라와 이상 이벤트, 정비 티켓을 확인하고 조치할 수 있습니다.

## 6. 시연할 때 보여줄 화면

시연에서는 다음 화면만 차례대로 보여주면 된다.

1. 영상 실행 화면
   - 준비한 `sample.mp4`를 실행한다.
   - 차량 탐지, OCR, 속도 계산이 되는 모습을 보여준다.
   - 설명 포인트: “이렇게 교통 데이터가 시스템으로 들어옵니다.”

2. 상태 샘플 주입
   - FPS 저하, 영상 흐림, 네트워크 지연 같은 상태 데이터를 넣는다.
   - 설명 포인트: “이제 카메라 상태가 나빠진 상황을 재현합니다.”

3. `/admin/ops` 관리자 화면
   - 위험 카메라 수를 확인한다.
   - 열린 이상 이벤트 수를 확인한다.
   - 생성된 정비 티켓 수를 확인한다.
   - 설명 포인트: “시스템이 문제를 감지해서 운영 화면에 표시했습니다.”

4. 이상 이벤트 목록
   - 어떤 문제가 감지됐는지 보여준다.
   - 예: FPS 저하, 프레임 드롭, 영상 흐림, 네트워크 불안정
   - 설명 포인트: “문제가 단순히 하나로 뭉쳐 보이는 것이 아니라 원인별로 나뉘어 표시됩니다.”

5. 정비 티켓 목록
   - 심각한 이상에 대해 티켓이 자동 생성된 것을 보여준다.
   - 설명 포인트: “운영자가 바로 조치할 수 있도록 티켓까지 만들어집니다.”

6. 티켓 상태 변경
   - 담당자 배정 또는 상태 변경 버튼을 클릭한다.
   - 설명 포인트: “감지에서 끝나는 것이 아니라 실제 운영 조치 흐름까지 연결됩니다.”

## 7. 현재 시연에서 기대하는 결과

현재 준비된 시연 기준으로는 다음 값이 보이면 정상이다.

- 위험 카메라: 1대
- 열린 이상 이벤트: 6건
- 정비 티켓: 3건

이상 이벤트 예시:

- FPS 저하
- 프레임 드롭
- 응답 지연
- 영상 흐림
- 리소스 과부하
- 네트워크 불안정

정비 티켓은 그중 심각도가 높은 문제에 대해 자동으로 생성된다.

## 8. 비개발자에게 설명하는 비유

비개발자에게는 병원 건강검진이나 자동차 경고등에 비유하면 쉽다.

예시 1:

> 자동차에 엔진 경고등이 뜨면 운전자가 바로 점검할 수 있습니다. 이 프로젝트는 CCTV와 AI 분석 시스템에 붙은 경고등 같은 역할을 합니다.

예시 2:

> 사람도 아프기 전에 건강검진을 해서 이상 징후를 찾습니다. 이 시스템도 카메라와 AI 분석 상태를 계속 검사해서 문제가 커지기 전에 알려줍니다.

예시 3:

> 단순히 CCTV 영상을 보는 시스템이 아니라, CCTV가 제대로 보고 있는지도 확인하는 시스템입니다.

## 9. 발표자가 외우면 좋은 핵심 문장

발표 중간에 사용할 수 있는 문장:

> 기존 기능은 차량을 탐지하는 것이고, 이번 기능은 그 차량 탐지 시스템이 안정적으로 동작하는지 관리하는 기능입니다.

> 카메라가 흐려지거나 FPS가 떨어지면 차량 인식 품질도 떨어질 수 있기 때문에, 시스템이 이런 상태를 자동으로 감지합니다.

> 심각한 이상은 단순 경고로 끝나지 않고, 정비 티켓으로 연결되어 운영자가 바로 조치할 수 있습니다.

> 이 기능을 통해 교통 분석 시스템을 실제 운영 환경에서 더 안정적으로 관리할 수 있습니다.

## 10. 발표 흐름 예시

아래 순서대로 말하면 자연스럽다.

1. “저희 기존 프로젝트는 CCTV 영상에서 차량을 탐지하고 번호판과 속도를 분석했습니다.”
2. “하지만 실제 운영에서는 카메라나 네트워크 상태가 나빠지면 분석 결과도 나빠질 수 있습니다.”
3. “그래서 이번에는 카메라와 AI 분석 시스템의 건강 상태를 감시하는 예지보전 기능을 추가했습니다.”
4. “먼저 샘플 영상을 넣어 교통 데이터가 들어오는 모습을 보여드리겠습니다.”
5. “이제 FPS 저하, 영상 흐림, 네트워크 지연 같은 상태 이상을 주입해보겠습니다.”
6. “관리자 화면을 보면 위험 카메라와 이상 이벤트가 자동으로 표시됩니다.”
7. “심각한 이상은 정비 티켓으로 자동 생성됩니다.”
8. “운영자는 이 티켓을 배정하고 처리 상태를 변경할 수 있습니다.”
9. “즉, 이 시스템은 차량 분석뿐 아니라 분석 시스템 자체의 안정성까지 관리합니다.”

## 11. 이 프로젝트의 가치

이 프로젝트의 가치는 다음처럼 정리할 수 있다.

- 장애를 사람이 늦게 발견하는 문제를 줄인다.
- 카메라 상태 저하를 조기에 발견할 수 있다.
- 차량 탐지와 OCR 결과의 신뢰도를 유지하는 데 도움이 된다.
- 운영자는 어떤 카메라를 먼저 봐야 하는지 쉽게 알 수 있다.
- 이상 감지에서 정비 티켓까지 이어져 실제 운영 흐름에 가깝다.

## 12. 발표에서 너무 깊게 말하지 않아도 되는 것

아래 내용은 최종 코드 학습자료에서 자세히 다루면 된다.

- Spring Boot 내부 service/controller 구조
- FastAPI 내부 detector adapter 구조
- DB table 상세 구조
- anomaly policy 세부 threshold
- transaction 처리 방식
- Docker healthcheck 세부 조건
- JWT 인증 구현 방식

발표에서는 다음 정도만 말하면 충분하다.

> Spring Boot는 운영 API와 DB 저장을 담당하고, FastAPI는 이상 감지를 담당합니다. Vue 화면에서는 그 결과를 운영자가 볼 수 있게 표시합니다.

## 13. 최종 한 줄 정리

이 프로젝트는 CCTV 교통 분석 시스템이 안정적으로 운영될 수 있도록 카메라와 AI 분석 상태를 감시하고, 이상 징후를 발견하면 정비 조치까지 연결해주는 예지보전 시스템이다.
