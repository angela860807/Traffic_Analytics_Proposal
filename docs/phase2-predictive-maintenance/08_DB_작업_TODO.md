## 1. 목표와 담당 파트

DB 파트는 ERD를 PostgreSQL 16 schema로 구현하고, 시계열·모델 예측·이벤트·정비 데이터를 중복 없이 저장하며 기준선 조회와 운영 API가 필요한 성능을 보장한다. Spring Java 코드는 수정하지 않는다.

| 역할 | 담당 파트 |
|---|---|
| 구현 소유 | DB 파트 |
| JPA mapping 협업 | Spring Boot 파트 |
| 분석 데이터 추출 협업 | AI 모델 파트 |

단독 소유 범위:

```text
backend/traffic/src/main/resources/db/migration/**
db/**
```

- [ ] `created_at` 기본값 `CURRENT_TIMESTAMP`
- [ ] `updated_at`은 Spring Boot가 변경 시 명시적으로 갱신
- [ ] SHADOW score·WARNING·CRITICAL threshold 0~1 check
- [ ] WARNING threshold < CRITICAL threshold
- [ ] model input window 시작 < 종료

활성 이벤트:

```sql
CREATE UNIQUE INDEX ux_anomaly_events_active
ON anomaly_events (target_camera_id, anomaly_type)
WHERE status IN ('OPEN', 'ACKNOWLEDGED', 'RECOVERED');
```

활성 이벤트 저장은 partial unique index를 최종 방어선으로 사용한다. Spring Boot는 충돌을 감지해 기존 이벤트를 조회·갱신하며, DB 파트는 동시 insert 테스트로 한 행만 활성 상태가 되는지 검증한다.

티켓 번호:

```sql
CREATE SEQUENCE maintenance_ticket_number_seq;
```

애플리케이션은 `nextval('maintenance_ticket_number_seq')`를 사용해 `MNT-YYYYMMDD-0001` 형식으로 만든다. sequence는 일자별 초기화하지 않으며 번호의 전역 유일성을 우선한다.

인덱스는 실제 API query와 `EXPLAIN ANALYZE` 결과를 근거로 추가한다. 모든 컬럼에 무분별하게 생성하지 않는다.

PostgreSQL 인덱스 정책:

- 기본 시계열 조회는 B-tree 복합 인덱스를 사용한다.
- 90일치 전체 시간 범위 검색이 많고 테이블이 커질 경우 `sampled_at` BRIN index를 비교한다.
- `JSONB`는 현재 API가 내부 key로 검색하지 않으므로 초기 GIN index를 만들지 않는다.
- bulk seed 후 `ANALYZE`를 실행하고 `pg_stat_user_indexes`로 미사용 인덱스를 확인한다.

## 6. 기준선 조회

`db/queries/camera_baseline.sql`:

- [ ] 카메라별 최근 14일
- [ ] 평가 시각과 동일한 30분 시간대
- [ ] `data_source='REAL'`
- [ ] `quality_status='COMPLETE'`
- [ ] `is_imputed=false`
- [ ] 활성·확정 장애 구간 제외
- [ ] 평가 시각 이후 데이터 제외
- [ ] metric별 median·MAD·sample count
- [ ] 표본 30개 미만 식별

동일 query는 `baselineDataSource` parameter를 받아 demo profile에서 `SIMULATED` 기준선을 조회할 수 있어야 한다. 한 기준선 안에서 출처를 혼합하지 않는다.

Spring Boot가 조회하기 쉬운 native query 또는 projection 결과 형태를 Spring Boot 파트와 확정한다.

PostgreSQL 중앙값·MAD 계산 예:

```sql
WITH filtered AS (
  SELECT fps_avg
  FROM camera_health_samples
  WHERE camera_id = :cameraId
    AND sampled_at >= :baselineFrom
    AND sampled_at < :evaluatedAt
    AND data_source = :baselineDataSource
    AND quality_status = 'COMPLETE'
    AND is_imputed = false
    AND fps_avg IS NOT NULL
),
median_value AS (
  SELECT percentile_cont(0.5)
         WITHIN GROUP (ORDER BY fps_avg) AS median
  FROM filtered
)
SELECT
  m.median,
  percentile_cont(0.5)
    WITHIN GROUP (ORDER BY abs(f.fps_avg - m.median)) AS mad,
  count(f.fps_avg) AS sample_count
FROM median_value m
LEFT JOIN filtered f ON true
GROUP BY m.median;
```

- `percentile_cont` 반환 타입과 Spring projection 타입을 통합 테스트한다.
- 평가 시각보다 미래인 행이 결과에 포함되지 않는 테스트를 둔다.
- 같은 30분 시간대 조건은 PostgreSQL session timezone에 의존하지 않도록 평가 시각을 기준으로 명시적으로 계산한다.

PostgreSQL batch delete 예:

```sql
DELETE FROM camera_health_samples
WHERE ctid IN (
  SELECT ctid
  FROM camera_health_samples
  WHERE sampled_at < :retentionCutoff
  LIMIT 10000
);
```

대량 삭제 후 즉시 `VACUUM FULL`을 실행하지 않는다. 일반 autovacuum 상태를 확인하고 필요한 경우 운영 시간에 `VACUUM (ANALYZE)`를 수행한다.

## 12. Migration 적용

- [ ] 적용 전 `pg_dump --schema-only` 백업
- [ ] `psql -v ON_ERROR_STOP=1` 사용
- [ ] `BEGIN ... ROLLBACK` 사전 검증
- [ ] 실제 적용 시 `BEGIN ... COMMIT`
- [ ] 적용 후 `\dt`, `\d+`, constraint·index 조회
- [ ] 이미 적용된 `001`~`006` 파일 checksum 또는 Git 변경 여부 확인

신규 테이블과 인덱스를 함께 만드는 이번 migration은 transaction 안에서 실행한다. 운영 중 큰 기존 테이블에 `CREATE INDEX CONCURRENTLY`가 필요해지면 transaction migration과 별도 파일로 분리한다.
