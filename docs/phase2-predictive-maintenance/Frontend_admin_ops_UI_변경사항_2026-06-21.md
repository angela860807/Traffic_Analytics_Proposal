# Frontend `/admin/ops` UI 변경사항 - 2026-06-21

## 1. 목적

시연 시 빔 프로젝트로 `/admin/ops`를 보여줄 때 운영자가 즉시 이해할 수 있도록 예지보전 상세 화면의 가독성을 개선했다.

핵심 목표:

- 영어 enum, metric, unit 노출 제거
- “무엇이 문제인지 / 어느 장비인지 / 어떤 조치를 해야 하는지 / 얼마나 위험한지”를 한 화면에서 확인
- 처리 버튼 중복 제거
- 상세는 판단 근거 확인, 현황표는 처리 작업 수행으로 역할 분리

## 3. 화면 용어 변경

### 3-1. 정비 용어

`티켓`은 비개발자 운영자에게 의미가 약하므로 화면 표시 용어를 `정비 건`으로 변경했다.

변경 예:

| 기존 | 변경 |
|---|---|
| 티켓 상태 | 정비 상태 |
| 미해결 티켓 | 미해결 정비 |
| 해결 티켓 | 완료 정비 |
| 연결된 정비 티켓 | 연결된 정비 건 |
| 티켓 상태 변경 | 정비 상태 변경 |

주의:

- API 함수명과 백엔드 계약은 `maintenanceTickets`, `ticketStatus`를 유지한다.
- 변경은 표시 계층에서만 적용했다.

### 3-2. 대응 시간 용어

`SLA 잔여`는 의미가 직관적이지 않아 `대응 제한시간`으로 변경했다.

변경 예:

| 기존 | 변경 |
|---|---|
| SLA 잔여 | 대응 제한시간 |
| SLA 초과만 | 대응 지연만 |
| 7분 초과 | 7분 지연 |
| 1.0시간 남음 | 1.0시간 남음 |

## 4. 우선 조치 이상 상세 개선

### 4-1. 상세 카드 역할 변경

기존 상단 상세 카드에는 다음 기능이 섞여 있었다.

- 이상 원인 확인
- 지표 확인
- 연결 정비 건 확인
- 해결
- 오탐 종료
- 담당자 배정

변경 후 역할:

- 상단 `우선 조치 이상 상세`: 원인과 조치 근거 확인 전용
- 하단 `이상 이벤트 및 정비 건 현황`: 배정, 작업 시작, 해결, 종결 처리

상단 카드에는 안내 문구를 추가했다.

```text
이 영역은 원인과 조치 근거 확인용입니다.
배정, 작업 시작, 해결, 종결 처리는 아래 현황표에서 진행하세요.
```

### 4-2. 선택 행 컨텍스트 추가

하단 현황표에서 `상세`를 클릭하면 상단 카드로 스크롤되고, 어떤 항목이 선택되었는지 표시한다.

표시 예:

```text
목록에서 선택한 이상
ANM-4 · Entry Camera 1 · 흐림 증가
연관 조치 지표: 영상 흐림 정도
```

선택 행은 연한 파란 배경으로 표시한다.

수정 이유:

- 기존에는 `상세` 클릭 후 상단으로 이동하지만 어떤 지표와 연결되는지 불명확했다.
- 선택 행 표시가 모든 셀에 파란 선을 만들어 보기 어색했으므로 첫 번째 셀 왼쪽 표시만 남겼다.

### 4-3. 상세 헤더 배지 구조 개선

기존 표시:

```text
심각 / 프레임 손실 증가 / 이상 탐지 / 티켓 대기 / 정책 룰 / P1
```

문제:

- 각 값이 무엇을 의미하는지 라벨이 없다.
- 시연 화면에서 단어가 나열된 것처럼 보인다.

변경 표시:

```text
위험도 심각
프레임 손실 증가
이상 상태 탐지
정비 상태 대기
탐지 방식 정책 룰
우선순위 P1
```

## 5. 조치 판단 지표 개선

### 5-1. 표에서 카드형 지표로 변경

기존:

- 지표명
- 관측값
- 기준선
- 임계값
- 판정

변경:

- 지표명
- 대상 장비
- 장비 영역
- 설명
- 권장 조치
- 현재
- 정상 범위
- 조치 기준
- 판정

### 5-2. 지표 한글화

예:

| API metric | 화면 표시 |
|---|---|
| `fps_avg` | 처리 FPS |
| `frame_drop_rate` | 프레임 손실률 |
| `latency_p95_ms` | 응답 지연시간 |
| `blur_score_avg` | 영상 흐림 정도 |
| `cpu_usage_pct` | CPU 사용률 |
| `memory_usage_pct` | 메모리 사용률 |
| `network_rtt_ms` | 네트워크 왕복 지연 |

### 5-3. 단위 정규화

예:

| API unit | 화면 표시 |
|---|---|
| `pct` | `%` |
| `ratio` | `%` |
| `rate` | `%` |
| `score` | `%` |
| `ms` | `밀리초` |
| `fps` | `프레임/초` |
| `count` | `건` |

비율 값 처리:

- `ratio`, `rate`: `0.18` -> `18%`
- `score`: `0.95` -> `95%`

### 5-4. 정상 범위 표시

기존에는 baseline이 없으면 `0%` 또는 `기준 없음`처럼 표시될 수 있었다.

변경:

- baseline이 있으면 baseline 값을 표시
- baseline이 없으면 정책 조치 기준으로 정상 범위를 표시

예:

| 지표 방향 | 표시 |
|---|---|
| 높을수록 위험 | `85% 이하` |
| 낮을수록 위험 | `5 프레임/초 이상` |

### 5-5. 위험 정도 표현

기존:

```text
임계 초과
```

변경:

```text
조치 필요
조치 기준보다 9%p 초과
```

FPS처럼 낮을수록 위험한 지표:

```text
조치 기준보다 2.0프레임/초 부족
```

### 5-6. 권장 조치 문구

예:

| 지표 | 권장 조치 |
|---|---|
| CPU 사용률 | AI 분석 프로세스 부하, 컨테이너 CPU 제한, 동시 스트림 수 확인 |
| 메모리 사용률 | 메모리 누수 또는 장시간 실행 프로세스 확인, 서비스 재시작/리소스 증설 검토 |
| 프레임 손실률 | 카메라 스트림 손실, 네트워크 품질, 프레임 버퍼 상태 확인 |
| 영상 흐림 정도 | 렌즈 오염, 초점 이탈, 흔들림과 조도 상태 현장 점검 |

## 6. 현황표 변경

### 6-1. 상태 분리

기존에는 이상 상태와 정비 상태가 섞여 보였다.

변경:

- `이상 상태`
- `정비 상태`

두 컬럼으로 분리했다.

### 6-2. 처리 작업 일원화

현황표에서 처리하는 작업:

- 배정
- 작업 시작
- 해결
- 최종 종결
- 상세 보기

상단 상세 카드의 중복 버튼은 제거했다.

### 6-3. 담당자 배정 후보 보정

문제:

- 기존 담당자 후보가 `7`, `8`, `9`, `12`로 하드코딩되어 있었다.
- 현재 Docker DB의 `members`에는 `1 이용자`, `2 관리자`만 존재했다.
- 그래서 `Assignee not found: 8` 오류가 발생했다.

변경:

- 기본 후보를 현재 DB에 존재하는 계정으로 조정
  - `2 관리자 (ADMIN)`
  - `1 이용자 (USER)`
- 모달 기본 선택값은 `관리자(id=2)`
